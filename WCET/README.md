## 1. Background

### 1.1 What is WCET

**WCET (Worst-Case Execution Time)** is one of the most important metrics in real-time systems, referring to the longest possible execution time a task may experience under worst-case conditions. In the ARTO project, accurate WCET estimation is critical for:

- **Task scheduling**: Ensuring all tasks complete before their deadlines
- **Integrity verification**: Distinguishing normal execution from abnormal delays
- **System design**: Determining safety margins and resource budgets

### 1.2 Why Extreme Value Analysis

Problems with traditional measurement methods (simply taking the maximum observed value):

- The observed maximum is not the true worst case
- Larger execution times may appear as the number of runs increases
- Extreme events are inherently low-probability events

**Extreme Value Analysis (EVA)** models the extremes (block maxima) in observed data to predict the worst-case execution time that a system may encounter over millions of runs. This tool uses the GEV (Generalized Extreme Value) distribution to fit block maxima.

### 1.3 GEV Distribution Parameters

```
GEV(c, loc, scale)
```

| Parameter | Meaning | Impact on WCET |
|-----------|---------|----------------|
| `c` (shape) | Tail behavior | `c > 0` → Fréchet (heavy tail, most dangerous); `c = 0` → Gumbel (exponential tail); `c < 0` → Weibull (bounded upper limit) |
| `loc` (location) | Center of extreme value distribution | Determines WCET baseline |
| `scale` (scale) | Dispersion of extremes | Larger → faster WCET growth |

## 3. Data Collection

### 3.1 Instrumentation Principle

In ArduPilot, execution time measurement code has been added to each major task function:

- Uses `AP_WCET::now_us()` to record task start/end times (microsecond timestamps)
- On **Linux SITL**, uses `CLOCK_THREAD_CPUTIME_ID` (measures CPU time only)
- On **embedded hardware**, uses `std::chrono::high_resolution_clock` (wall-clock)
- Each task's measurements are written in batches to `timing_log.txt` via the `TimingLogger` class
- Flushed to disk every 256 executions or on program exit

> **Key improvement**: Originally used `std::chrono::high_resolution_clock::now()` (wall-clock), which under SITL was contaminated by OS scheduling preemption. Now changed to `AP_WCET::now_us()`, which uses `CLOCK_THREAD_CPUTIME_ID` on Linux, measuring only the CPU time actually consumed by the thread, excluding scheduling noise.

### 3.3 Data Format

Each line in `timing_log.txt`:

```
<task_name> <execution_time_us>
```

Example:


Each line represents the duration (in microseconds) of a single execution of that task. All executions of all tasks are recorded interleaved in chronological order.

## 4. Extreme Value Analysis

### 4.1 Basic Usage

Enter the `WCET/` directory and run directly (initial artifact filtering enabled by default):

```bash
cd ARTO/WCET/
python3 wcet_eva.py --input timing_log.txt
```

Initial artifact filtering is enabled by default (`--no-filter` to disable). Output is written to the `results/` directory.

### 4.2 Initial Artifact Filtering

Before the train/test split, the script filters out measurement contamination points caused by instrumentation overhead such as file writes and frequent `print` calls, on a per-task basis. These points are excluded from both the training set and the test set. Filtering strategy:

1. Compute **Q1** (25th percentile) and **Q3** (75th percentile) for the task's execution times
2. Compute the interquartile range **IQR = Q3 - Q1**
3. Compute a configurable high percentile **P<tail-percentile>** (default P95)
4. Compute MAD (Median Absolute Deviation) as a robust scale measure
5. Use the upper bound: **threshold = max(Q3 + 1.5 × IQR, P<tail-percentile>, MAD-z upper bound)**
6. Remove measurements exceeding the `threshold`

This strategy is more lenient than the standard Tukey fence, aiming to remove isolated instrumentation contamination spikes while preserving the true WCET right tail as much as possible.

Filtering information is displayed in the output:

```
  [filter] Q1=118.00, Q3=141.00, IQR=23.00, P95=312.00, MAD=8.00, threshold=312.00 (max of Q3+1.5×IQR, P95, MAD-z), removed 1/98560 outliers (max kept=312)
```

For data without scheduling noise contamination (e.g., rc_loop), no filtering is triggered:

```
  rc_loop: max <= threshold → no filtering triggered
```

Use `--no-filter` to disable filtering:

```bash
python3 wcet_eva.py --input timing_log.txt --no-filter
```

Adjust initial artifact filtering strength:

```bash
python3 wcet_eva.py --input timing_log.txt --tail-percentile 95
python3 wcet_eva.py --input timing_log.txt --artifact-iqr-multiplier 1.5 --artifact-mad-z 8
```

### 4.3 Block Size Adjustment

The script defaults to generating approximately 40 block maxima per task:

```bash
python3 wcet_eva.py --input timing_log.txt
```

If block maxima are too concentrated after filtering, or if fitting is unstable, increase the target number of blocks to make each block smaller and produce more extreme points:

```bash
python3 wcet_eva.py --input timing_log.txt --target-blocks 80
python3 wcet_eva.py --input timing_log.txt --target-blocks 100
```

You can also specify the block size directly. The script uses a synthetic 1-second index, so the number of seconds here is equivalent to the sample count per block:

```bash
python3 wcet_eva.py --input timing_log.txt --block-seconds 500
```

It is recommended to compare results at `40`, `80`, and `100` blocks. If WCET estimates vary significantly, the task's extreme value fit is sensitive to block size, requiring more samples or more cautious interpretation.

### 4.4 Automated Block Size and Confidence Level Tuning

To systematically determine block size and confidence level, use `--tune` mode:

```bash
python3 wcet_eva.py --input timing_log.txt --output tune_results --tune
```

Default search range:

- Block size: `25..300`
- Block step: default `1`, use `--block-step 5` for faster coarse search
- Confidence level alpha: `97..99`, step `0.2`
- Data split: 80% train/calibration, 20% test; the train/calibration portion is further split with 25% for calibration

Explicit specification:

```bash
python3 wcet_eva.py --input timing_log.txt --output tune_results \
  --tune \
  --block-min 25 --block-max 300 --block-step 5 \
  --alpha-min 97 --alpha-max 99 --alpha-step 0.2
```

Tuning process:

1. For each task, split data chronologically into train/calibration/test sets
2. Fit GEV with different block sizes on the fit subset of the training set
3. Select block size using the calibration set: prefer candidates whose calibration set does not exceed the point-estimate WCET; if all exceed, choose the candidate with the lowest exceedance rate
4. After fixing block size, select the confidence level on the alpha grid `97..99`: prefer the narrowest candidate whose calibration set does not exceed the confidence interval upper bound
5. Refit the final model using the full training set
6. Evaluate the final WCET only on the test set, reporting the number of test set exceedances of the point estimate and the confidence interval upper bound

If the block maxima for a task are too degenerate within the specified block size range for GEV to be statistically meaningful, the script uses an empirical fallback: selecting the block size with the most stable calibration set performance and using the training set maximum as a conservative WCET, marked as `fallback` in the report.

Output files:

- **`wcet_tuning_summary.md`** — Best block size, alpha, and test set exceedance counts per task
- **`wcet_tuning_results.csv`** — More complete tuning metrics, suitable for plotting or paper tables

### 4.5 Paper-Style Parameter Selection Plots

To generate parameter selection plots following the paper's experimental procedure:

```bash
python3 wcet_eva.py --input timing_log.txt --output param_select_results \
  --experiment-param-select
```

This mode:

1. Splits each real task chronologically into 70% fit set, 30% validation set
2. Let `k` denote the number of execution time samples per block
3. Start from `k=40`, increment by `10`
4. Extract block maxima from each task's fit set and fit GEV using MLE
5. Use the chi-squared goodness-of-fit test; `p-value > 0.05` is considered a successful fit for that task at that `k`
6. Count the proportion of successfully fitted tasks at different `k` values, and generate `gev_k_fit_ratio.png`
7. After selecting a unified `k`, test `alpha ∈ {0.99, 0.995, 0.999, 0.9995, 0.9999}`
8. If all validation set samples do not exceed `WCET_alpha = GEV.ppf(alpha)`, the task passes coverage validation
9. Count the proportion of tasks passing coverage validation at different `alpha` values, and generate `gev_alpha_coverage_ratio.png`

Configurable parameter example:

```bash
python3 wcet_eva.py --input timing_log.txt --output param_select_results \
  --experiment-param-select \
  --experiment-seed 42 \
  --experiment-k-min 40 --experiment-k-max 300 --experiment-k-step 10 \
  --experiment-alphas 0.99,0.995,0.999,0.9995,0.9999
```

Output files:

- **`gev_k_fit_ratio.png`** — Proportion of tasks successfully fitting GEV at different `k`
- **`gev_alpha_coverage_ratio.png`** — Proportion of tasks whose validation set is covered by WCET at different `alpha`
- **`k_fit_results.csv`** — Chi-squared test results per task, per `k`
- **`alpha_coverage_results.csv`** — Validation coverage results per task, per `alpha`
- **`param_selection_summary.md`** — Selected unified `k` and `alpha`

### 4.6 Per-Task Independent k and Alpha Selection

If a unified `k` is not required and you want each task to have its own `k` and `alpha`:

```bash
python3 wcet_eva.py --input timing_log.txt --output per_task_params \
  --per-task-param-table \
  --experiment-seed 42 \
  --experiment-k-min 40 --experiment-k-max 300 --experiment-k-step 10 \
  --experiment-alphas 0.99,0.995,0.999,0.9995,0.9999
```

This mode performs independently for each task:

1. Splits each task chronologically into 70% fit set, 30% validation set
2. Selects the smallest `k` from candidates with chi-squared test `p-value > 0.05`
3. If no `k` passes, selects the `k` with the highest `p-value` and marks it as `warning`
4. Fixing that task's `k`, selects the smallest `alpha` from candidates that covers all validation set samples
5. Outputs each task's `k`, `p-value`, `alpha`, `WCET_alpha`, and validation set exceedance info

Output files:

- **`per_task_k_alpha_table.md`** — Paper-style table format
- **`per_task_k_alpha_table.csv`** — Machine-readable table
- **`per_task_k_fit_details.csv`** — Chi-squared test details per task, per candidate `k`

## 5. Interpreting Results

### 5.1 Summary Table

```
| Task   | Samples | Filtered | Observed Max (μs) | Model | 100-run WCET (μs) | 1M-run WCET (μs) |
|--------|---------|----------|-------------------|-------|-------------------|------------------|
| rc_loop| 32,768  |          | 527               | MLE   | 798               | 1,038            |
```

- **Samples**: Number of observed samples for this task (after filtering)
- **Filtered**: ✓ indicates initial artifacts were filtered
- **Observed Max**: Measured maximum after filtering
- **100-run WCET**: Maximum execution time likely to occur per 100 blocks
- **1M-run WCET**: Maximum execution time likely to occur per 1,000,000 blocks

### 5.2 Diagnostic Plots

The diagnostic plot contains four subplots:

1. **Trace** — Time series + marked block maxima
2. **Histogram** — Extreme value distribution histogram
3. **Q-Q** — Quantile comparison; points falling near the diagonal indicate good fit
4. **Return Level** — Return level plot

### 5.3 Typical Result Analysis

**Good behavior — Stable tasks**

```
| update_land_and_crash_detectors_update | 32,760 | ✓ | 59  | MLE | 242 | 377 |
| ekf_check                               | 256    |   | 15  | MLE | 34  | 45  |
| ten_hz_logging_loop                     | 768    |   | 335 | MLE | 1,052 | 1,474 |
```

These tasks show gentle WCET growth, with only 2-5× increase from observed maximum to 1M-run prediction. GEV fitting is stable.

**Significant improvement from filtering — Tasks contaminated by scheduling noise**

| Task | 1M-run before filtering (μs) | 1M-run after filtering (μs) | Improvement factor |
|------|------------------------------|-----------------------------|-------------------|
| AP_InertialSensor6update | 155,684,478 | **120** | ~1,300,000× |
| GCS14update_receive | 202,841 | **359** | ~565× |
| GCS14update_send | 10,871,649 | **311,121** | ~35× |

`AP_InertialSensor6update` is one of the tasks more noticeably contaminated by instrumentation or scheduling noise. Extreme spikes in the raw maximum may originate from file writes, frequent `print` calls, or OS preemption. After initial artifact filtering, the predicted values better reflect the task's true CPU execution time distribution.

### 5.4 Assessing Fit Quality

1. **Q-Q plot**: Points should roughly follow the diagonal
2. **Log-likelihood**: Larger values (smaller absolute negative values) generally indicate better fit
3. **c parameter (shape)**:
   - `c` close to 0: Gumbel distribution, suitable for most soft real-time tasks
   - `c < 0`: Weibull distribution, WCET has a theoretical upper bound
   - `c > 0`: Fréchet distribution, heavy tail, requires attention
4. **Confidence intervals**: The shaded region in the return level plot should not be excessively wide

## 6. FAQ

### Q1: Why do I need to run SITL to collect data?

`timing_log.txt` is generated by ArduPilot SITL simulation and reflects the execution time distribution of actual flight control tasks under typical workloads. The `timing_log.txt` in the `WCET/` directory currently contains data from a single simulation run.

### Q2: Why are WCET predictions abnormally high before filtering?

This is because the current SITL environment runs on Linux, where `std::chrono::high_resolution_clock::now()` measures wall-clock time. When the Linux scheduler preempts the ArduPilot process to run other system services, the timer does not stop, causing some measurements to include tens of milliseconds of scheduling delay.

```
Original measurement:
  t1 = clock_gettime(CLOCK_MONOTONIC)   ← wall-clock
  ... task execution ... [OS preemption 5ms]
  t2 = clock_gettime(CLOCK_MONOTONIC)
  duration = t2 - t1  ← includes 5ms scheduling delay!
```

Solutions:
- **Solution 1** (code level): Use `CLOCK_THREAD_CPUTIME_ID` (already implemented in `AP_WCET::now_us()`), measuring only the CPU time actually consumed by the thread
- **Solution 2** (data analysis level): Use a combined threshold of IQR, high percentile, and MAD to filter instrumentation contamination points before the train/test split (already implemented in `wcet_eva.py`)

### Q3: How do I collect more data for more accurate WCET?

Run longer simulations with more diverse missions (takeoff, hover, landing, waypoint flight), keep the file in append mode, and merge data from multiple runs into `timing_log.txt`.

### Q4: How many samples are enough?

- At least 200+ samples per task
- 500+ samples yield reasonably stable GEV fitting
- For WCET predictions at 1,000,000 runs, at least several thousand samples are recommended

### Q5: Why are WCET predictions still high for some tasks after filtering?

A few tasks (e.g., `update_fight_mode`, `read_mode_switch`) still show high WCET predictions after filtering. This is typically because:

1. **The task itself has high execution time variance** (not scheduling noise)
2. **Insufficient sample size** (only 40 blocks)
3. **Unstable GEV fitting** (Weibull distribution with c value near singularity)

Recommendation: Focus on 100/1000 block predictions and ignore the extreme 1M block predictions.

### Q6: Can these issues be avoided on real hardware?

When running ArduPilot on real hardware (STM32 + ChibiOS):
- **No OS preemption**: ChibiOS is an RTOS without Linux-style scheduling preemption
- **Precise timers**: Hardware timers directly measure CPU execution time
- Therefore, neither Solution 1 nor Solution 2 corrections are needed

The current corrections specifically address measurement contamination in the SITL simulation environment.
